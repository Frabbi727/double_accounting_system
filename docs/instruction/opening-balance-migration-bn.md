# ওপেনিং ব্যালেন্স মাইগ্রেশন আর্কিটেকচার

> পুরনো ব্যবসার হিসাব নতুন সিস্টেমে আনার সম্পূর্ণ নকশা। এইটা একবারই হবে — তাই একবারেই ঠিক হতে হবে।

---

## ভাগ ০ — কেন এই ধাপটাই সবচেয়ে ঝুঁকিপূর্ণ

দৈনিক বিক্রয়ে ভুল হলে আজই ধরা পড়ে — কাস্টমার বলবে "ভাই হিসাব ভুল"। কিন্তু ওপেনিং ব্যালেন্সে ভুল হলে:

- আজ ধরা পড়বে না
- আগামী মাসে ধরা পড়বে না
- ৬ মাস পর যখন লাভের হিসাব মিলবে না, তখন খুঁজতে গিয়ে পাবেন **সব রিপোর্ট ভুল ছিল শুরু থেকেই**
- ততদিনে হাজার হাজার লেনদেন এন্ট্রি হয়ে গেছে, ঠিক করা প্রায় অসম্ভব

তাই এই ধাপে **তাড়াহুড়া করবেন না**। আর্কিটেকচার এমন হতে হবে যেন ভুল হলে **সহজে ঠিক করা যায়**।

---

## ভাগ ১ — সবচেয়ে বড় আর্কিটেকচারাল সিদ্ধান্ত

আগের ডকুমেন্টে আমি লিখেছিলাম `accounts.opening_balance` কলাম রাখতে। **সেটা ভুল ছিল, এবং এখন সংশোধন করছি** — কারণ আপনার ক্ষেত্রে পুরনো ব্যবসার পুরো হিসাব আনতে হবে।

### ❌ যেভাবে করবেন না — আলাদা কলাম

```php
accounts:  id | name | opening_balance    ← এখানে ২৫,০০০ বসানো
```

**সমস্যা কী:**

```
ক্যাশ ব্যালেন্স = opening_balance কলাম + জার্নাল থেকে হিসাব
                  ↑                        ↑
              এক জায়গা                আরেক জায়গা
```

হিসাব দুই জায়গা থেকে আসছে। ফলে:

- ট্রায়াল ব্যালেন্স মিলবে না (ওপেনিং তো জার্নালে নেই!)
- ওপেনিং ভুল হলে সরাসরি কলাম এডিট করতে হবে — কোনো ট্রেইল থাকবে না
- "৳২৫,০০০ কোথা থেকে এলো?" — উত্তর দেওয়ার কোনো রেকর্ড নেই
- ব্যালেন্স শিট তৈরি করা যাবে না, কারণ Owner's Equity কোথাও নেই

### ✅ যেভাবে করবেন — ওপেনিংও একটা জার্নাল এন্ট্রি

```
কোনো opening_balance কলাম নেই।
ওপেনিং ব্যালেন্স = একটা বিশেষ জার্নাল এন্ট্রি, তারিখ = cut-off date
```

```
Journal Entry #1 — "Opening Balance as on 31-07-2026"
┌────────────────────────────────┬──────────────┬──────────────┐
│ অ্যাকাউন্ট                      │    Debit     │    Credit    │
├────────────────────────────────┼──────────────┼──────────────┤
│ 1010  ক্যাশ ইন হ্যান্ড           │    ২৫,০০০    │              │
│ 1021  ডাচ-বাংলা ব্যাংক          │  ১,৫০,০০০    │              │
│ 1040  ইনভেন্টরি                 │  ১,২০,০০০    │              │
│ 1030  কাস্টমার বাকি (AR)        │     ৮,০০০    │              │
│ 2010  সাপ্লায়ার বাকি (AP)       │              │    ৪৫,০০০    │
│ 2020  ব্যাংক লোন                │              │  ২,০০,০০০    │
│ 3010  মালিকের পুঁজি (Equity)    │              │    ৫৮,০০০    │
├────────────────────────────────┼──────────────┼──────────────┤
│                          মোট   │  ৩,০৩,০০০    │  ৩,০৩,০০০    │ ✅ মিলে গেছে
└────────────────────────────────┴──────────────┴──────────────┘
```

**এতে কী কী সুবিধা:**

| সুবিধা | ব্যাখ্যা |
|---|---|
| এক সত্য | ব্যালেন্স শুধু জার্নাল থেকে আসে, দুই জায়গা নেই |
| ট্রায়াল ব্যালেন্স মেলে | ওপেনিং সহ পুরো হিসাব একসাথে যাচাই হয় |
| Equity অটো বের হয় | ৩,০৩,০০০ − ২,৪৫,০০০ = ৫৮,০০০ (সিস্টেম নিজে হিসাব করবে) |
| সংশোধনযোগ্য | ভুল হলে রিভার্সাল + নতুন এন্ট্রি, ট্রেইল থাকে |
| একই কোড | ওপেনিং আর দৈনিক লেনদেন — দুটোই `LedgerService::post()` ব্যবহার করবে |

> **মূল কথা:** ওপেনিং ব্যালেন্স কোনো "বিশেষ জিনিস" না। এটা শুধু একটা জার্নাল এন্ট্রি যার তারিখ সবার আগে। আলাদা কোড, আলাদা কলাম, আলাদা নিয়ম — কিছুই লাগবে না।

## ভাগ ২ — একই নিয়ম স্টকের জন্য

স্টকেও একই ভুল হয়। `products.opening_qty` কলাম রাখবেন না।

### ✅ ওপেনিং স্টক = একটা `stock_movement`

```
stock_movements
┌────┬──────────┬──────┬─────┬───────────┬────────────────┬────────────┐
│ id │ product  │ type │ qty │ unit_cost │ reference_type │    date    │
├────┼──────────┼──────┼─────┼───────────┼────────────────┼────────────┤
│  1 │ সাবান     │  in  │  ৫০ │    ৪০.০০  │ OpeningBalance │ ৩১-০৭-২৬  │
│  2 │ শ্যাম্পু    │  in  │  ৩০ │    ৯৫.০০  │ OpeningBalance │ ৩১-০৭-২৬  │
└────┴──────────┴──────┴─────┴───────────┴────────────────┴────────────┘
```

স্টক হিসাব একই কুয়েরিতে হবে — ওপেনিং আলাদা করে ধরার দরকার নেই।

**গুরুত্বপূর্ণ যাচাই:** স্টক মুভমেন্টের মোট মূল্য = জার্নালের ইনভেন্টরি ডেবিট

```
(৫০ × ৪০) + (৩০ × ৯৫) = ২,০০০ + ২,৮৫০ = ৳ ৪,৮৫০
                                              ↑
                            জার্নালে 1040 ইনভেন্টরি ডেবিট এই সংখ্যাই হতে হবে
```

এই দুইটা না মিললে **মাইগ্রেশন বাতিল করবেন**। সিস্টেম নিজে এই চেক করবে।

## ভাগ ৩ — কাস্টমার ও সাপ্লায়ার বাকি (এখানে একটা সূক্ষ্ম ব্যাপার আছে)

জার্নালে আপনি লিখলেন "কাস্টমার বাকি ৳৮,০০০"। কিন্তু এই ৮,০০০ কার কার কাছে?

তাই **দুই স্তরে** রাখতে হবে:

```
স্তর ১ — কন্ট্রোল অ্যাকাউন্ট (জার্নালে)
  1030  Accounts Receivable    Debit  ৳ ৮,০০০

স্তর ২ — পার্টি সাবসিডিয়ারি (আলাদা টেবিলে)
  করিম ভাই      ৳ ৫,০০০
  রহিম স্টোর     ৳ ২,০০০
  জসিম          ৳ ১,০০০
  ─────────────────────
  মোট           ৳ ৮,০০০   ← এইটা স্তর ১-এর সমান হতেই হবে
```

### কীভাবে রাখবেন

প্রতিটা পার্টির ওপেনিং বাকিও একটা লেনদেন হিসেবে রাখুন — যেন লেজারে দেখা যায়:

```php
// opening_party_balances টেবিল
Schema::create('opening_party_balances', function (Blueprint $table) {
    $table->id();
    $table->morphs('party');                    // Customer বা Supplier
    $table->date('date');                        // cut-off date
    $table->decimal('amount', 15, 2);
    $table->string('reference')->nullable();     // পুরনো ইনভয়েস নং
    $table->date('original_date')->nullable();   // মূল বিক্রয়ের তারিখ ★
    $table->foreignId('journal_entry_id')->constrained();
    $table->timestamps();
});
```

> **`original_date` কেন জরুরি?** বাকির বয়স (Aging) রিপোর্টের জন্য। করিম ভাইয়ের ৳৫,০০০ যদি ৬ মাস আগের হয়, তাহলে সেটা "৯০+ দিন" ক্যাটাগরিতে দেখাতে হবে — cut-off তারিখ থেকে ০ দিন না। এই ফিল্ডটা না রাখলে aging রিপোর্ট অর্থহীন হয়ে যাবে।

**আরো ভালো উপায় (যদি পুরনো ইনভয়েসের বিস্তারিত থাকে):** প্রতিটা বকেয়া ইনভয়েস আলাদা রো হিসেবে ঢোকান। তাহলে "কোন ইনভয়েসের টাকা বাকি" পর্যন্ত ট্র্যাক করা যাবে, আর কাস্টমার পেমেন্ট করলে নির্দিষ্ট ইনভয়েসের সাথে মেলানো (allocation) যাবে।

### ধাপ ১ — Staging টেবিল

আসল টেবিলে সরাসরি ঢোকাবেন না। আগে staging-এ:

```php
Schema::create('migration_staging_products', function (Blueprint $table) {
    $table->id();
    $table->string('batch_id');              // একই মাইগ্রেশনের সব রো
    $table->string('source_ref')->nullable(); // পুরনো সিস্টেমের আইডি
    $table->string('name');
    $table->string('unit')->nullable();
    $table->decimal('qty', 15, 3)->nullable();
    $table->decimal('cost_price', 15, 2)->nullable();
    $table->decimal('sale_price', 15, 2)->nullable();
    $table->json('errors')->nullable();       // যাচাইয়ের ভুলগুলো
    $table->enum('status', ['pending','valid','invalid','imported'])->default('pending');
    $table->timestamps();
});
```

একই ধরনের staging টেবিল থাকবে: `_customers`, `_suppliers`, `_accounts`, `_party_balances`।

**সুবিধা:** ভুল ডেটা পেলে staging-এ ঠিক করে আবার চালাবেন। আসল ডেটা কখনো নোংরা হবে না।

### ধাপ ২ — যাচাইয়ের নিয়ম

```php
class OpeningBalanceValidator
{
    public function validate(string $batchId): ValidationReport
    {
        $errors = collect();

        // — গঠনগত যাচাই —
        $errors = $errors->merge($this->checkDuplicateNames($batchId));
        $errors = $errors->merge($this->checkNegativeQuantities($batchId));
        $errors = $errors->merge($this->checkMissingCostPrice($batchId));
        $errors = $errors->merge($this->checkZeroCostWithStock($batchId));

        // — যুক্তিগত যাচাই —
        $errors = $errors->merge($this->checkSalePriceBelowCost($batchId));   // সতর্কতা
        $errors = $errors->merge($this->checkFutureDates($batchId));
        $errors = $errors->merge($this->checkPartyBalanceSigns($batchId));

        // — হিসাবগত যাচাই ★ সবচেয়ে জরুরি —
        $errors = $errors->merge($this->checkInventoryValueMatchesJournal($batchId));
        $errors = $errors->merge($this->checkArSubsidiaryMatchesControl($batchId));
        $errors = $errors->merge($this->checkApSubsidiaryMatchesControl($batchId));
        $errors = $errors->merge($this->checkJournalBalances($batchId));

        return new ValidationReport($errors);
    }
}
```

**যে যাচাইগুলো অবশ্যই থাকতে হবে:**

| যাচাই | কেন |
|---|---|
| স্টক আছে কিন্তু ক্রয়মূল্য ০ | লাভের হিসাব সম্পূর্ণ ভুল আসবে |
| ইনভেন্টরি মোট = জার্নাল ডেবিট | দুই দিক না মিললে ব্যালেন্স শিট ভাঙবে |
| কাস্টমার মোট = AR কন্ট্রোল | সাবসিডিয়ারি আর কন্ট্রোল মিলতেই হবে |
| সাপ্লায়ার মোট = AP কন্ট্রোল | একই কারণ |
| জার্নাল Debit = Credit | না মিললে কিছুই কাজ করবে না |
| একই নামে দুই পণ্য | পরে গুলিয়ে যাবে |
| ঋণাত্মক স্টক | অসম্ভব, ডেটা ভুল |
| ভবিষ্যতের তারিখ | cut-off-এর পরে হতে পারে না |

### ধাপ ৩ — Dry Run

সবচেয়ে গুরুত্বপূর্ণ ধাপ। **হিসাব করে দেখাবেন, কিন্তু সেভ করবেন না:**

```php
public function dryRun(string $batchId): OpeningBalancePreview
{
    return DB::transaction(function () use ($batchId) {
        $result = $this->buildOpeningEntry($batchId);
        DB::rollBack();          // ★ সব বাতিল, শুধু হিসাবটা ফেরত দিলাম
        return $result;
    });
}
```

### ধাপ ৪ — পোস্টিং

```php
public function commit(string $batchId, int $approvedBy): OpeningBalanceResult
{
    return DB::transaction(function () use ($batchId, $approvedBy) {

        // ১. মাস্টার ডেটা তৈরি
        $products  = $this->importProducts($batchId);
        $customers = $this->importCustomers($batchId);
        $suppliers = $this->importSuppliers($batchId);
        $accounts  = $this->importAccounts($batchId);

        // ২. স্টক মুভমেন্ট
        $inventoryValue = $this->createOpeningStockMovements($batchId, $products);

        // ৩. পার্টি ব্যালেন্স
        $arTotal = $this->createOpeningPartyBalances($batchId, 'customer');
        $apTotal = $this->createOpeningPartyBalances($batchId, 'supplier');

        // ৪. Equity হিসাব (ব্যালেন্সিং ফিগার)
        $assets      = $cashTotal + $bankTotal + $inventoryValue + $arTotal;
        $liabilities = $apTotal + $loanTotal;
        $equity      = $assets - $liabilities;

        // ৫. মূল জার্নাল এন্ট্রি
        $entry = app(LedgerService::class)->post(
            date: $this->cutOffDate,
            referenceType: 'OpeningBalance',
            referenceId: $batchId,
            description: "Opening Balance as on {$this->cutOffDate}",
            lines: $lines                    // ★ post() নিজেই Debit=Credit যাচাই করবে
        );

        // ৬. চূড়ান্ত যাচাই — না মিললে পুরোটা বাতিল
        $this->assertTrialBalanced();
        $this->assertSubsidiariesMatchControls();

        // ৭. লক
        $this->lockOpeningBalance($entry, $approvedBy);

        return new OpeningBalanceResult($entry, $equity);
    });
}
```

## ভাগ ৫ — ভুল হলে কীভাবে ঠিক করবেন

পোস্ট করার পর ভুল ধরা পড়লে **কখনো ডিলিট করবেন না**। রিভার্সাল দিবেন:

```
১. মূল ওপেনিং এন্ট্রি          → অপরিবর্তিত থাকল
২. রিভার্সাল এন্ট্রি            → উল্টো ডেবিট-ক্রেডিট, একই তারিখ
3. সংশোধিত ওপেনিং এন্ট্রি      → ঠিক সংখ্যা নিয়ে নতুন এন্ট্রি
```

```php
public function correctOpeningBalance(array $correctedData, string $reason): void
{
    DB::transaction(function () use ($correctedData, $reason) {
        $this->periodService->unlock($this->cutOffDate, auth()->id(), $reason);

        $original = JournalEntry::where('reference_type', 'OpeningBalance')->firstOrFail();
        $this->ledger->reverse($original, "সংশোধন: {$reason}");

        $this->commit($correctedData, auth()->id());
        // commit() নিজেই আবার লক করবে
    });
}
```

**স্টকের ক্ষেত্রেও একই** — `stock_movements`-এ বিপরীত মুভমেন্ট দিয়ে বাতিল করে নতুন মুভমেন্ট দিবেন।

## ভাগ ৬ — ক্রয়মূল্য নিয়ে একটা সতর্কতা

পুরনো ব্যবসা থেকে আসা স্টকের ক্রয়মূল্য প্রায়ই ঠিকমতো জানা থাকে না। এইটা নিয়ে সিদ্ধান্ত নিতে হবে:

| পরিস্থিতি | কী করবেন |
|---|---|
| ক্রয়মূল্য জানা আছে | সেটাই দিন — সবচেয়ে ভালো |
| জানা নেই, কিন্তু সাম্প্রতিক ক্রয় বিল আছে | শেষ ক্রয়মূল্য ব্যবহার করুন |
| কিছুই নেই | বিক্রয়মূল্য থেকে আনুমানিক মার্জিন বাদ দিন (যেমন ২৫%) |
| ❌ কখনো করবেন না | ক্রয়মূল্য ০ রাখা |

**ক্রয়মূল্য ০ রাখলে কী হয়:** ওই পণ্য বিক্রি করলে COGS = ০ হবে, মানে সিস্টেম দেখাবে **পুরো বিক্রয়টাই লাভ**। আপনার লাভের রিপোর্ট আকাশছোঁয়া দেখাবে, ট্যাক্সের হিসাবও ভুল হবে।

তাই ভ্যালিডেটরে এটা **hard error** রাখুন — ক্রয়মূল্য ০ থাকলে মাইগ্রেশন এগোবেই না।

## ভাগ ৮ — যাচাইয়ের চেকলিস্ট (পোস্ট করার পর)

মাইগ্রেশনের পর এই ৮টা মিলিয়ে দেখুন। **সব মিললে তবেই দৈনিক এন্ট্রি শুরু করবেন:**

- [ ] ট্রায়াল ব্যালেন্স: মোট Debit = মোট Credit
- [ ] ব্যালেন্স শিট: সম্পদ = দায় + পুঁজি
- [ ] স্টক রিপোর্টের মোট মূল্য = জার্নালের ইনভেন্টরি ব্যালেন্স
- [ ] সব কাস্টমারের বাকির যোগফল = AR কন্ট্রোল অ্যাকাউন্ট
- [ ] সব সাপ্লায়ারের বাকির যোগফল = AP কন্ট্রোল অ্যাকাউন্ট
- [ ] ক্যাশ ব্যালেন্স = আসলে বাক্সে যত টাকা আছে (গুনে দেখুন!)
- [ ] ব্যাংক ব্যালেন্স = ব্যাংক স্টেটমেন্টের সাথে মেলে
- [ ] পুঁজির অংক মালিকের হিসাবের সাথে যুক্তিসঙ্গত

শেষ তিনটা **হাতে কলমে মেলাতে হবে** — কোড দিয়ে হবে না। ক্যাশ বাক্স খুলে টাকা গুনুন, ব্যাংকের স্টেটমেন্ট বের করুন।

## ভাগ ৯ — সংশোধিত সিদ্ধান্ত তালিকা

আগের আর্কিটেকচার ডকুমেন্টের যে অংশগুলো **বদলে গেল**:

| বিষয় | আগে বলেছিলাম | ★ এখন যা করবেন | কেন |
|---|---|---|---|
| অ্যাকাউন্টের ওপেনিং | `accounts.opening_balance` কলাম | জার্নাল এন্ট্রি | এক সত্য, ট্রায়াল ব্যালেন্স মেলে |
| পণ্যের ওপেনিং স্টক | `products.opening_qty` কলাম | `stock_movements` রেকর্ড | একই কারণ |
| পার্টির ওপেনিং বাকি | `customers.opening_balance` কলাম | `opening_party_balances` + জার্নাল | Aging রিপোর্ট সম্ভব হয় |
| Owner's Equity | ধরা হয়নি | ব্যালেন্সিং ফিগার হিসেবে অটো | ব্যালেন্স শিট সম্পূর্ণ হয় |
| পিরিয়ড লক | ছিল না | `accounting_periods` টেবিল | ওপেনিং দুর্ঘটনাক্রমে বদলাবে না |
| মাইগ্রেশন প্রক্রিয়া | ছিল না | staging → validate → dry-run → commit | ভুল ধরা পড়ে সেভ হওয়ার আগেই |

## পরের ধাপ

আর্কিটেকচার এখন আপনার পরিস্থিতির জন্য প্রস্তুত। কোড শুরু করার আগে আমার দুইটা প্রশ্ন আছে:

**১. পুরনো ডেটা কোন অবস্থায় আছে?** — খাতায় লেখা / এক্সেল ফাইল / পুরনো সফটওয়্যার (কোনটা)? এর উপর ইম্পোর্টার কেমন হবে সেটা নির্ভর করে।

**২. পুরনো বকেয়ার ইনভয়েস-ভিত্তিক বিস্তারিত আছে কি?** — অর্থাৎ "করিম ভাই ৳৫,০০০" জানেন, নাকি "করিম ভাই — ইনভয়েস #১২৩ ৳৩,০০০ (১৫ মার্চ) + #১৪৫ ৳২,০০০ (২ এপ্রিল)" পর্যন্ত জানেন? বিস্তারিত থাকলে aging রিপোর্ট অনেক ভালো হবে।

উত্তর পেলে migration + `LedgerService` + `OpeningBalanceService` — এই তিনটার সম্পূর্ণ কোড লিখে দিব।

# ফর্মের ভেতরেই ওপেনিং ব্যালেন্স — নকশা

> আপনি যা চান: কাস্টমার/সাপ্লায়ার/পণ্য তৈরির সময় ওই ফর্মেই ওপেনিং দিয়ে দিবেন। আলাদা কোনো ধাপ নেই।
> ভেতরে যা হবে: প্রতিটা ওপেনিং নিজে থেকে জার্নাল এন্ট্রি হয়ে যাবে — যাতে হিসাব কখনো না ভাঙে।

## ভাগ ২ — জার্নালের অপর দিক কী হবে?

একটা জার্নাল এন্ট্রিতে সবসময় দুই দিক লাগে। ওপেনিং ব্যালেন্সের **অপর দিক সবসময় হবে `3010 — মালিকের পুঁজি (Owner's Equity)`**।

কারণ: এই সম্পদ বা দায় আপনার পুরনো ব্যবসা থেকে এসেছে, নতুন কোনো লেনদেন থেকে নয়। তাই এটা আপনার পুঁজির অংশ।

| যা যোগ করছেন | Debit | Credit | পুঁজিতে প্রভাব |
|---|---|---|---|
| কাস্টমার ওপেনিং বাকি ৳৫,০০০ | কাস্টমার বাকি (AR) | মালিকের পুঁজি | পুঁজি ৫,০০০ বাড়ল |
| সাপ্লায়ার ওপেনিং বাকি ৳৪৫,০০০ | মালিকের পুঁজি | সাপ্লায়ার বাকি (AP) | পুঁজি ৪৫,০০০ কমল |
| পণ্য ওপেনিং স্টক ৳২,০০০ | ইনভেন্টরি | মালিকের পুঁজি | পুঁজি ২,০০০ বাড়ল |
| ক্যাশ অ্যাকাউন্ট ওপেনিং ৳২৫,০০০ | ক্যাশ | মালিকের পুঁজি | পুঁজি ২৫,০০০ বাড়ল |
| ব্যাংক লোন ওপেনিং ৳২,০০,০০০ | মালিকের পুঁজি | ব্যাংক লোন | পুঁজি ২,০০,০০০ কমল |

**সুন্দর ব্যাপারটা কী:** আপনি যত খুশি একটা একটা করে যোগ করতে থাকুন — যেকোনো মুহূর্তে ট্রায়াল ব্যালেন্স মিলবে, আর পুঁজির অংক নিজে থেকে ঠিক জায়গায় গিয়ে দাঁড়াবে। আগে থেকে সব হিসাব করে রাখার দরকার নেই।

## ভাগ ৪ — কোড কেমন হবে

### ৪.১ একটামাত্র সার্ভিস সব সামলাবে

```php
namespace App\Services\Accounting;

class OpeningEntryService
{
    public function __construct(
        private LedgerService $ledger,
        private PeriodLockService $periodLock,
    ) {}

    /**
     * যেকোনো মাস্টার রেকর্ডের ওপেনিং ব্যালেন্স পোস্ট করে।
     * কাস্টমার, সাপ্লায়ার, পণ্য, অ্যাকাউন্ট — সবার জন্য এই একটাই।
     */
    public function post(
        Account $account,        // কোন অ্যাকাউন্টে যাবে (AR/AP/Inventory/Cash)
        float $amount,
        string $date,
        Model $source,           // Customer/Supplier/Product/Account মডেল
        ?string $reference = null,
    ): JournalEntry {

        $equity = Account::code('3010')->firstOrFail();   // মালিকের পুঁজি

        // অ্যাকাউন্টের ধরন দেখে ডেবিট না ক্রেডিট, সিস্টেম নিজে ঠিক করবে
        $lines = $account->type->increasesWithDebit()
            ? [
                ['account_id' => $account->id, 'debit' => $amount, 'credit' => 0],
                ['account_id' => $equity->id,  'debit' => 0,       'credit' => $amount],
              ]
            : [
                ['account_id' => $equity->id,  'debit' => $amount, 'credit' => 0],
                ['account_id' => $account->id, 'debit' => 0,       'credit' => $amount],
              ];

        return $this->ledger->post(
            date: $date,
            referenceType: 'Opening',
            referenceId: $source->id,
            description: $this->describe($source, $reference),
            lines: $lines,
        );
    }
}
```

> **এইটাই মূল সৌন্দর্য:** কাস্টমার হোক বা পণ্য — একই সার্ভিস, একই কোড। অ্যাকাউন্টের ধরন (asset নাকি liability) দেখে সিস্টেম নিজেই ঠিক করে নেয় ডেবিট হবে না ক্রেডিট। আপনাকে মনে রাখতে হবে না।

### ৪.২ কাস্টমার তৈরির সময়

```php
class CustomerService
{
    public function create(array $data): Customer
    {
        return DB::transaction(function () use ($data) {

            $customer = Customer::create([
                'name'          => $data['name'],
                'phone'         => $data['phone'] ?? null,
                'credit_limit'  => $data['credit_limit'] ?? 0,
                'default_discount' => $data['default_discount'] ?? 0,
            ]);

            // ওপেনিং বাকি দেওয়া থাকলে
            if (!empty($data['opening_balance']) && $data['opening_balance'] > 0) {

                $entry = $this->openingEntry->post(
                    account:   Account::code('1030')->first(),   // কাস্টমার বাকি
                    amount:    $data['opening_balance'],
                    date:      $data['opening_date'] ?? config('shop.cutoff_date'),
                    source:    $customer,
                    reference: $data['opening_reference'] ?? null,
                );

                // aging রিপোর্টের জন্য বিস্তারিত রাখুন
                OpeningPartyBalance::create([
                    'party_type'       => Customer::class,
                    'party_id'         => $customer->id,
                    'amount'           => $data['opening_balance'],
                    'original_date'    => $data['opening_date'],   // ★ মূল তারিখ
                    'reference'        => $data['opening_reference'],
                    'journal_entry_id' => $entry->id,
                ]);
            }

            return $customer;
        });
    }
}
```

### ৪.৩ পণ্য তৈরির সময়

```php
class ProductService
{
    public function create(array $data): Product
    {
        return DB::transaction(function () use ($data) {

            $product = Product::create([...]);

            if (!empty($data['opening_qty']) && $data['opening_qty'] > 0) {

                $cost  = $data['opening_cost'] ?? $data['cost_price'];
                $value = $data['opening_qty'] * $cost;

                // ১. স্টক মুভমেন্ট
                StockMovement::create([
                    'product_id'     => $product->id,
                    'type'           => MovementType::In,
                    'qty'            => $data['opening_qty'],
                    'unit_cost'      => $cost,
                    'reference_type' => 'Opening',
                    'reference_id'   => $product->id,
                    'date'           => $data['opening_date'] ?? config('shop.cutoff_date'),
                ]);

                // ২. জার্নাল এন্ট্রি — একই সার্ভিস
                $this->openingEntry->post(
                    account: Account::code('1040')->first(),   // ইনভেন্টরি
                    amount:  $value,
                    date:    $data['opening_date'] ?? config('shop.cutoff_date'),
                    source:  $product,
                );
            }

            return $product;
        });
    }
}
```

সাপ্লায়ার আর অ্যাকাউন্টের কোডও হুবহু একই প্যাটার্নে — শুধু অ্যাকাউন্ট কোড আলাদা (`2010` সাপ্লায়ার, `1010`/`1020` ক্যাশ-ব্যাংক)।

## ভাগ ৫ — যেসব সমস্যা হতে পারে, আর তার সমাধান

আপনি বলেছেন "যেন কোনো সমস্যা না হয়" — তাই সম্ভাব্য প্রতিটা সমস্যা আর তার প্রতিরোধ ধরে ধরে বলছি।

### সমস্যা ১: ভুল করে ওপেনিং-এ চলতি লেনদেন ঢুকে যাওয়া

> ৩ মাস পর ইউজার নতুন কাস্টমার যোগ করল আর ওপেনিং বাকিতে আজকের বিক্রির টাকা বসিয়ে দিল।

**প্রতিরোধ:** cut-off তারিখের পরে ওপেনিং দেওয়া বন্ধ করে দিন।

```php
// FormRequest-এ
'opening_date' => [
    'nullable',
    'date',
    'before_or_equal:' . config('shop.cutoff_date'),
],
'opening_balance' => [
    'nullable',
    'numeric',
    'min:0',
    function ($attr, $value, $fail) {
        if ($value > 0 && $this->periodLock->isLocked(config('shop.cutoff_date'))) {
            $fail('ওপেনিং ব্যালেন্সের সময়কাল লক করা আছে। '
                . 'নতুন বাকি যোগ করতে চাইলে বিক্রয় এন্ট্রি দিন।');
        }
    },
],
```

ইউজারকে পরিষ্কার বার্তা দিবেন: *"এখন আর ওপেনিং দেওয়া যাবে না। টাকা পাওনা হলে বিক্রয় এন্ট্রি দিন।"*

### সমস্যা ২: ওপেনিং দেওয়ার পর এডিট করতে চাওয়া

> ইউজার কাস্টমার এডিট করে ওপেনিং ৫,০০০ থেকে ৭,০০০ করে দিল।

**প্রতিরোধ:** এডিট ফর্মে ওপেনিং ফিল্ড **read-only** করে দিন, পাশে আলাদা বাটন:

```
│  ওপেনিং বাকি:  ৳ ৫,০০০  🔒                       │
│  (১৫/০৩/২০২৬ তারিখে পোস্ট করা)                   │
│  [ সংশোধন করুন ]  ← আলাদা অনুমতি লাগবে            │
```

সংশোধনে ক্লিক করলে রিভার্সাল + নতুন এন্ট্রি হবে, সরাসরি এডিট নয়:

```php
public function correctOpening(Customer $customer, float $newAmount, string $reason): void
{
    $this->authorize('correctOpeningBalance', $customer);   // শুধু admin

    DB::transaction(function () use ($customer, $newAmount, $reason) {
        $old = $customer->openingBalance;

        $this->ledger->reverse($old->journalEntry, "সংশোধন: {$reason}");
        $old->update(['reversed_at' => now(), 'reversal_reason' => $reason]);

        $this->openingEntry->post(
            account: Account::code('1030')->first(),
            amount:  $newAmount,
            date:    $old->original_date,
            source:  $customer,
        );
    });
}
```

### সমস্যা ৩: ক্রয়মূল্য ছাড়া স্টক ঢোকানো

> ইউজার তাড়াহুড়ায় ক্রয়মূল্য ০ রেখে ৫০ পিস স্টক দিল।

**ফলাফল:** ওই পণ্য বিক্রি করলে COGS = ০, সিস্টেম দেখাবে পুরো বিক্রয়টাই লাভ। রিপোর্ট সম্পূর্ণ ভুল।

**প্রতিরোধ:** hard validation, কোনো ছাড় নেই।

```php
'opening_cost' => [
    'required_if:has_opening_stock,true',
    'numeric',
    'gt:0',   // ★ ০ চলবে না
],
```

ফর্মে লাইভ সতর্কতা দেখান: *"⚠ ক্রয়মূল্য ছাড়া স্টক দিলে লাভের হিসাব ভুল আসবে"*

### সমস্যা ৪: একই কাস্টমার দুইবার ঢোকানো

### সমস্যা ৬: ওপেনিং দেওয়ার আগেই বিক্রয় এন্ট্রি

> স্টক ঢোকানোর আগেই কেউ বিক্রয় করতে গেল — negative stock হয়ে যাবে।

**প্রতিরোধ:** ওপেনিং লক না হওয়া পর্যন্ত দৈনিক লেনদেন বন্ধ রাখুন।

```php
// Middleware
class RequireOpeningBalanceComplete
{
    public function handle($request, Closure $next)
    {
        if (!$this->openingService->isComplete()) {
```

### সমস্যা ৭: `Opening` জার্নাল অনেকগুলো আলাদা হয়ে যাওয়া

> ২৩ জন কাস্টমার = ২৩টা আলাদা জার্নাল এন্ট্রি। রিপোর্টে জটিল লাগবে।

**সমাধান:** এটা আসলে সমস্যা নয়, বরং **সুবিধা** — প্রতিটা আলাদা থাকায় একটা ভুল হলে শুধু সেটাই রিভার্স করা যায়, বাকিগুলো ছুঁতে হয় না।

রিপোর্টে দেখানোর সময় `reference_type = 'Opening'` দিয়ে গ্রুপ করে একলাইনে দেখান:

```php
$openingTotal = JournalEntryLine::whereHas('journalEntry',
    fn($q) => $q->where('reference_type', 'Opening')
)->sum('debit');
```

## পরের ধাপ

আর্কিটেকচার এখন সম্পূর্ণ। কোড লেখা শুরু করার জন্য আমার শুধু একটা জিনিস জানা দরকার:

**আপনার পুরনো ডেটা কোথায় আছে?** — খাতায় লেখা, এক্সেল ফাইল, নাকি পুরনো কোনো সফটওয়্যারে?

- **খাতায়** → ফর্মে হাতে টাইপ করবেন, উপরের নকশাই যথেষ্ট
- **এক্সেলে** → একটা বাড়তি "এক্সেল থেকে আপলোড" সুবিধা যোগ করব (একই সার্ভিস ব্যবহার করবে, শুধু ইনপুট আলাদা)
- **পুরনো সফটওয়্যারে** → ডেটা কী ফরম্যাটে বের করা যায় জানলে সেই অনুযায়ী ইম্পোর্টার লিখব

উত্তর পেলে migrations + `LedgerService` + `OpeningEntryService` + কাস্টমার/পণ্য ফর্ম — সব কোড ধাপে ধাপে লিখে দিব।
